# Message Platform 项目架构文档

> **项目定位**：多源消息采集、存储、检索平台，支持从不同消息源采集新闻、学术论文等内容，并提供统一的检索接口和可视化大屏展示。

---

## 📋 项目总览

### 核心功能
- 27个国际消息源自动采集（RSS、网页爬取）
- 智能去重与增量更新
- 向量化存储与语义检索
- 多语言翻译与摘要生成（5种语言）
- AI字段增强（地区、行业、AI标签）
- 定时任务调度与监控
- 3D地球可视化大屏

### 技术架构
```
技术栈：
├── 后端：FastAPI + SQLAlchemy + Celery
├── 数据库：MySQL 8.0 + ChromaDB + Redis
├── 前端：Vue 3 + TypeScript + Element Plus
├── 采集：Playwright + feedparser + BeautifulSoup4
├── AI服务：LLM API（OpenAI/通义千问）
└── 任务调度：Celery Beat
```

#### 核心服务说明

**CollectorService（采集器调度服务）**
- 作用：管理所有消息源的采集器实例，触发采集任务
- 特性：动态加载采集器模块、浏览器池注入、错误隔离

**BrowserPool（浏览器池服务）**
- 作用：共享浏览器实例，降低资源消耗
- 特性：动态扩缩容（0-10实例）、健康检查、空闲回收

**FieldEnricherService（字段增强服务）**
- 作用：使用LLM分析消息内容，提取地区、行业标签、AI标签
- 输入：消息content
- 输出：`{region, industry_tags, ai_tag}`

**AIReportGenerator（AI日报生成服务）**
- 作用：每日定时生成AI领域日报（6个类型）
- 类型：治理、研究、产业、综合、中国AI、上海周报

### `collectors/` - 采集器基础设施
**作用**：提供统一的采集器基类  
**结构**：
```
collectors/
└── base/
    └── playwright_collector_base.py  # Playwright采集器基类（222行）
```
**基类职责**：
- 浏览器生命周期管理（从池获取/独立启动）
- 定时循环执行`_collect_once()`
- 错误隔离（单次采集失败不影响后续）

**子类实现规范**（三阶段）：
```python
class MyCollector(PlaywrightCollectorBase):
    async def _collect_once(self):
        # 阶段1：Scraping（数据抓取）
        articles = await self._scrape_articles()
        
        # 阶段2：Processing（预处理，在DB会话外）
        for article in articles:
            article['translated'] = await translator.translate(article['content'])
            article['enriched'] = await field_enricher.enrich(article['content'])
        
        # 阶段3：Storing（存储，仅DB操作）
        with create_session() as db:
            for article in articles:
                db.add(MyMessage(**article))
            db.commit()
```

## 🔄 数据流向

### 1. 消息采集流程
```
定时触发（Celery Beat）
  ↓
Celery Worker接收任务
  ↓
CollectorService调度采集器
  ↓
PlaywrightCollectorBase执行_collect_once()
  ├── 阶段1：Scraping（抓取原始数据）
  ├── 阶段2：Processing（翻译+字段增强，DB会话外）
  └── 阶段3：Storing（并发写入MySQL + ChromaDB）
  ↓
向量同步服务（VectorSync）
  ├── 全量同步：所有记录
  └── 增量同步：最近1000条
  ↓
前端实时更新（SSE推送）
```

### 2. AI日报生成流程
```
定时触发（Celery Beat，每天20:00）
  ↓
Celery Worker（report队列）接收任务
  ↓
AIReportGenerator执行
  ├── 从ChromaDB检索相关消息（向量检索）
  ├── 使用Chat模型（qwen3-max）生成日报
  └── 保存到mp_ai_daily_reports表
  ↓
前端查询展示
```

### 3. 消息检索流程
```
用户输入查询（前端）
  ↓
POST /api/v1/messages/search
  ↓
SearchService执行
  ├── 使用Embedding模型向量化查询
  ├── ChromaDB向量检索（语义匹配）
  └── 结合MySQL筛选（时间、来源）
  ↓
返回结果（Markdown渲染）
```

## 🗄️ 数据库架构

### MySQL表结构（核心表）

#### 1. `mp_message_sources` - 消息源配置表
**作用**：存储27个消息源的配置  
**核心字段**：
```sql
id               VARCHAR(36)    # UUID主键
name             VARCHAR(200)   # 消息源名称
display_name     VARCHAR(200)   # 显示名称
category         VARCHAR(50)    # 分类（news/academic/blog）
config           JSON           # 配置（mysql_table, collector_module, rss_url等）
is_active        TINYINT(1)     # 是否启用
interval         INT            # 采集间隔（秒）
schedule         VARCHAR(100)   # Cron表达式
last_crawled_at  DATETIME       # 最后采集时间
```

#### 2. 消息表（27个，统一字段标准）
**作用**：存储各消息源的消息数据  
**核心字段（10个必备）**：
```sql
id               VARCHAR(36)    # UUID主键
source_id        VARCHAR(36)    # 外键→mp_message_sources
external_id      VARCHAR(100)   # 外部唯一标识
title            VARCHAR(500)   # 标题
content          TEXT           # 正文内容
summary          TEXT           # 摘要
provider         VARCHAR(500)   # 作者/信息提供方
published_at     DATETIME       # 发布时间
crawled_at       DATETIME       # 抓取时间
url              VARCHAR(500)   # 原文链接
```
**扩展字段（可选）**：
```sql
region           VARCHAR(200)   # 地区（如"中国/广东省"）
industry_tags    TEXT           # 行业标签（逗号分隔）
ai_tag           VARCHAR(50)    # AI相关标签
language         VARCHAR(10)    # 语言代码
```

#### 3. `mp_ai_daily_reports` - AI日报表
**作用**：存储AI日报数据  
**核心字段**：
```sql
id               VARCHAR(36)    # UUID主键
report_type      VARCHAR(50)    # 日报类型（governance/research等）
report_date      DATE           # 日报日期
title            VARCHAR(500)   # 标题
content          TEXT           # 内容（Markdown格式）
metadata         JSON           # 元数据（消息来源、Token统计等）
created_at       DATETIME       # 生成时间
```

### ChromaDB集合结构
**作用**：向量存储，支持语义检索  
**集合命名**：每个消息源一个collection（如`bloomberg_tech`）  
**元数据字段**：
```json
{
  "id": "业务唯一标识",
  "source_id": "消息源ID",
  "title": "标题",
  "summary": "摘要",
  "published_at": "发布时间",
  "url": "原文链接"
}
```

## 🔑 关键特性

### 1. 配置驱动架构
- 所有消息源配置存储在数据库（mp_message_sources表）
- 采集器、ORM模型自动注册（零硬编码）
- 新增消息源仅需3步：创建采集器 → 注册SQL → 激活

### 2. 三阶段数据流（Scraping → Processing → Storing）
- 严格分离数据抓取、预处理、存储逻辑
- 避免在DB会话内执行异步IO（防止连接池耗尽）
- 并发写入MySQL和ChromaDB

### 3. 浏览器池优化
- 动态扩缩容（0-10实例）
- 空闲超时回收（5分钟）
- 健康检查与自动替换

### 4. 多模型LLM架构
- Chat模型（qwen3-max）：复杂推理
- Embedding模型（bge-m3）：向量化
- Fast模型（GLM-4.5-Air）：轻量任务

### 5. 智能去重
- 多级降级策略（URL → external_id → title）
- Sliding Window增量采集（遇到latest_url立即停止）

### 6. AI日报定时生成
- 6种日报类型（治理、研究、产业、综合、中国AI、上海周报）
- 每天20:00自动生成
- 向量检索相关消息 + LLM生成

### 1. 监控指标
- 采集器状态（运行/停止/错误）
- Token使用量（Chat/Embedding/Fast）
- 消息数量趋势（MySQL + ChromaDB）
- 浏览器池状态（活跃/空闲实例数）

### 2. 日志位置
- FastAPI日志：控制台输出
- Celery Worker日志：控制台输出
- ChromaDB日志：./data/chromadb/chroma.log

### 3. 数据备份
```bash
# MySQL备份
mysqldump -u root -p message_platform > backup_$(date +%Y%m%d).sql

# ChromaDB备份
tar -czf chromadb_backup_$(date +%Y%m%d).tar.gz ./data/chromadb
```

### 4. 常见问题

**Q1：采集器采集失败**
- 检查消息源配置（config字段）
- 检查浏览器池状态
- 查看Celery Worker日志

**Q2：向量同步失败**
- 检查ChromaDB连接（端口11530）
- 检查MySQL数据是否存在
- 运行`verify_vector_consistency.py`脚本

**Q3：AI日报生成失败**
- 检查LLM API配置
- 检查Token余额
- 查看report队列Worker日志
