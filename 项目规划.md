# Message Platform 项目规划

## 一、项目概述

Message Platform 是一个多源消息采集、存储、检索平台，支持从不同消息源（RSS、网页爬取、API）采集新闻、学术论文等内容，并提供统一的检索接口。

### 1.1 核心功能

- 多消息源自动采集（定时任务）
- 智能去重与增量更新
- 向量化存储与语义检索
- 多语言翻译与摘要生成
- 字段增强（地区、行业标签、AI标签）

### 1.2 技术栈

- 后端框架: FastAPI
- 数据库: MySQL 8.0
- ORM: SQLAlchemy
- 向量数据库: ChromaDB
- 翻译服务: LLM API（OpenAI/Claude）
- 采集库: feedparser, BeautifulSoup4, Playwright

### 3.2 核心服务模块

#### 3.2.1 CollectorService（采集器调度服务）

**职责**：
- 管理所有消息源的采集器实例
- 根据interval/schedule配置触发采集任务
- 动态加载采集器模块

**关键方法**：
- `register_collector(source_id, collector_class)`: 注册采集器
- `start_all()`: 启动所有启用的采集器
- `stop_all()`: 停止所有采集器

#### 3.2.2 FieldEnricherService（字段增强服务）

**职责**：
- 使用LLM分析消息内容，提取地区信息
- 识别行业标签（最多3个）
- 判断是否与AI相关

**输入**：消息content
**输出**：
```python
{
    "region": "中国/广东省",
    "industry_tags": "人工智能,半导体,自动驾驶",
    "ai_tag": "ai_related"
}
```

#### 3.2.3 Translator（翻译服务）

**职责**：
- 将外文内容翻译成中文
- 支持多语言（英语、德语、法语）
- 降级策略：翻译失败时返回截断原文

**幻觉检测**：
- 检测模板占位符：`{{variable}}`
- 检测引导式回复："请提供"、"我将为您"
- 检测元描述："以下是翻译结果"

## 四、采集器架构

### 4.1 统一基类架构（2025年强制要求）

**核心原则**：所有新采集器必须继承PlaywrightCollectorBase

**基类位置**：backend/collectors/base/playwright_collector_base.py（222行）

**架构层次图**：
```
PlaywrightCollectorBase（基类）
├── 浏览器生命周期管理
│   ├── initialize(): 从浏览器池acquire或独立启动
│   ├── run(): 定时循环执行_collect_once()
│   └── stop(): 释放浏览器到池或关闭独立浏览器
├── 浏览器池集成
│   ├── set_browser_pool(): CollectorService自动注入
│   └── self._browser: 子类通过此访问浏览器
└── 错误隔离：单次采集失败不影响后续循环

MyCollector（子类）
├── __init__(config):
│   ├── super().__init__(config)  # 必须调用
│   └── 初始化服务（translator, field_enricher, chroma_storage）
├── _on_initialize(): ChromaDB初始化等（可选）
└── _collect_once(): 单次采集核心逻辑（必须实现）
    ├── Scraping: 使用self._browser创建页面，抓取数据
    ├── Processing: 数据库会话外翻译+字段增强
    └── Storing: 仅数据库CRUD，不调用外部服务
```

**已迁移采集器（11/26）**：
- venturebeat, nikkei_asia, techcrunch, techinasia
- tonghuashun, kr36, securities_times
- partnership_ai, govai, gcg_ai, korea_herald

**待迁移采集器（15个）**：主要为RSS采集器和部分简单网页爬取采集器

### 4.2 三阶段数据流（在_collect_once内实现）

**Scraping → Processing → Storing 严格分离**：

**阶段1：Scraping（数据抓取）**
- 使用`self._browser.new_page()`创建页面
- 滚动加载、遇到latest_url立即停止
- 返回原始数据字典列表

**阶段2：Processing（预处理）**
- 在数据库会话**外部**执行
- 调用`translator.translate()`翻译
- 调用`field_enricher.enrich_fields()`增强字段
- 批内串行处理（Translator内部有并发控制）

**阶段3：Storing（存储）**
- 仅使用`with create_session() as db:`
- 不包含任何await外部服务
- 使用`asyncio.gather()`并发写MySQL和ChromaDB

**为什么重要**：异步IO在数据库事务内执行会长时间占用连接池，导致连接耗尽

### 4.3 Golden Example参照标准

**PlaywrightCollectorBase新架构最佳实践**：backend/sources/venturebeat/collector.py
- 继承PlaywrightCollectorBase
- __init__初始化服务（translator, field_enricher, chroma_storage）
- _on_initialize创建ChromaDB collection
- _collect_once实现三阶段：scrape → preprocess → store
- 通过self._browser访问浏览器（基类自动管理）

**增量采集优化**：backend/sources/nikkei_asia/collector.py
- 同样继承PlaywrightCollectorBase
- _scrape_articles逐页检查，遇到latest_url立即停止
- Fail Fast配置验证

**反面教材（禁止模仿）**：backend/sources/cigi/collector.py
- 未迁移到新架构（仍手动管理Playwright）
- _store_to_mysql在数据库会话内调用await translator
- 违反异步编程铁律

### 4.3 去重策略

**多级降级策略**：
1. 优先使用url去重（UNIQUE约束）
2. 降级使用external_id
3. 最后降级使用title
4. 启动时查询最新URL，遇到已存在记录立即停止（Sliding Window）

## 五、API接口设计

### 5.1 RESTful标准

**原则**：
- 后端直接返回数据对象，不包装成`{success, data}`格式
- 使用HTTP状态码表示成功/失败
- DELETE端点返回HTTP 204 No Content

### 5.2 核心端点

#### 5.2.1 消息源管理

```
GET    /api/v1/sources              # 获取消息源列表
GET    /api/v1/sources/{source_id}  # 获取单个消息源
POST   /api/v1/sources              # 创建消息源
PUT    /api/v1/sources/{source_id}  # 更新消息源
DELETE /api/v1/sources/{source_id}  # 删除消息源（CASCADE删除消息）
```

#### 5.2.2 消息查询

```
GET    /api/v1/messages                    # 获取消息列表
GET    /api/v1/messages/{message_id}       # 获取单条消息
POST   /api/v1/messages/search             # 向量语义搜索
GET    /api/v1/sources/{source_id}/messages # 获取特定消息源的消息
```

## 六、ChromaDB向量存储

### 6.1 集合管理

**命名规范**：
- 集合名称从消息源config.chroma_collection字段读取
- 每个消息源对应一个独立集合

### 6.2 向量化标准

**ID使用规则**：
- 使用业务唯一标识（seq、item_id、arxiv_id）
- 不使用uuid.uuid4()动态生成

**元数据字段**：
- id, source_id, title, summary, published_at, url

### 6.3 分批处理

- 每批50条，防止超时
- 使用BATCH_SIZE常量
